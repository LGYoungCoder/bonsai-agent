# Bonsai Philosophy

## 为什么要做这个

Bonsai 想成为**只属于你的 agent**：7×24 长期跑、多通道都能接入、所有数据落在你自己的盘上、用你自己的 API key。

要让它真能这样跑下去，必须把三件事当作硬约束：

- **Token 预算是真实成本** —— 你自己付钱，每个 token 都看得见。前缀字节稳定吃满 cache，工具返回截断 + 落盘，历史 thinking 优先压。
- **写入纪律高于写入便利** —— 跑半年的 agent 经不起记忆污染。公理 "No Execution, No Memory"：只有真正成功的工具调用结果才能进长期记忆，模型的猜测、推理、"我觉得应该"一律不写。
- **索引有损，原文无损** —— Verbatim 存储，绝不改写。索引层可以压到极致，原文层永远 100% 可召回。

剩下的所有设计决定，都是这三条的推论。

## 四条核心观察

### 观察一：token 不是无限的

在 30K 上下文里跑出 200K 上下文才能做到的效果是可能的，前提是把 token 当作硬约束管理：

- 每轮 LLM 调用前估算上下文预算
- 工具返回大输出就截断 + 存文件
- 历史压缩：把旧轮次的 `<thinking>` / `<tool_use>` / `<tool_result>` 内部截断，保留结构
- 工具 schema 固定前缀字节稳定，保 prompt cache

**结论**：**token 预算要当硬约束管，不能"先写功能再说"**。

### 观察二：幻觉污染长期记忆是致命的

公理 "No Execution, No Memory"：**只有成功的工具调用结果才能写入长期记忆**。模型的推理、猜测、"我觉得应该"一律不能写。

这条纪律让 `memory/` 目录在用了几个月之后仍然干净——没有那种"模型编造的虚假 SOP"。

**结论**：**写入纪律 > 写入便利**。

### 观察三：Verbatim 才能 100% 召回

信条：**逐字存储，绝不改写**。传统 RAG 的摘要链路会在每一次压缩时丢信息，叠加几次之后就面目全非。把"索引"和"原文"物理分开，索引可以随便压，原文永远在那儿。

纯向量搜索 + verbatim 原文，在 LongMemEval 这类长期记忆基准上能跑出 96%+ R@5。索引层有损压缩省 token，原文层永远无损。

**结论**：**索引有损，原文无损；两者必须分开存储**。

### 观察四：把保存记忆这件事从 chat window 挪走

Agent 主循环里**只做读，不做写**。写入全部走后台（subprocess / hook 事件触发）。这条的 token 节省是**线性叠加**的——session 越长，省得越多。

> *重新传输一次完整日记上下文的成本可能高达每 session 一美元；把写入搬到后台后，这部分直接归零。*

**结论**：**持久化不该出现在 chat 里**。

## Bonsai 的四条核心原则

### 1. 约束优先（Constraint First）

任何新功能上线前，先问："它占多少 token？"。没有 token 预算就不写代码。

**硬红线**：
- 每轮固定开销（sys + tools + L1）≤ 3K tokens
- 工具数 ≤ 12
- Wake-up ≤ 1K tokens
- 单次工具返回 ≤ 20K 字符（大输出转存文件）

破红线的 PR 不合并，理由不论。

### 2. 双库分离（Dual-Store）

两种记忆**必须**分开：

- **SkillStore**（distilled）：Agent 能力库。SOP、脚本、避坑规则。关键词检索。几十到几百条。
- **MemoryStore**（verbatim）：用户历史库。对话、事实、偏好。向量 + BM25 混合检索。几千到几万条。

绝不试图用一套 embedding 解决两种问题。理由：技能记忆关心"怎么做"（程序化），对话记忆关心"说过什么"（事实性），两者的检索模式、写入模式、更新模式完全不同。

### 3. 后台持久化（Background Persistence）

Agent 主循环**只读**。所有写入操作：
- L4 会话归档
- SkillStore 的 SOP 提炼
- MemoryStore 的 drawer 入库
- 索引更新

**全部走异步子进程或外部 hook**，零 chat token。

### 4. 索引金字塔（Pyramid Index）

所有记忆按"金字塔"组织：

```
塔尖 (≤1K tokens)        ← 每次对话常驻
  ↓ 压缩的指针
塔腰 (按需加载)          ← 模型主动 read
  ↓ 原文指针
塔基 (无上限)            ← 语义搜索/文件系统
```

**上层只存"能定位到下层的最短标识"**，多一个词都是冗余。

## 不做的事

以下是**刻意不做**的决定，请 PR 前先看一遍：

- ❌ **插件市场**：增加复杂度。技能通过 SkillStore 自然积累即可。
- ❌ **多租户 / 多用户**：Bonsai 是**单用户专属**工具。多租户场景请用别的。
- ❌ **摘要式记忆**：一切写入要么 verbatim（MemoryStore），要么 distilled-with-evidence（SkillStore）。没有第三种。
- ❌ **云端同步**：本地优先。同步自己用 rsync / git。
- ❌ **工具自动注册**：工具数量是硬约束。加工具必须走 PR。
- ❌ **"聪明"的上下文管理**：用规则、用预算、用硬裁。不引入新的 LLM 调用去管理 LLM 调用。

## 个人 Agent 的信任模型

Bonsai 的安全设计**和企业 agent 完全不同**。三条公理:

1. **运行 Bonsai 的机器 = 用户自己的机器**。Agent 的系统权限 = 用户的系统权限
2. **Agent 的使用者 = 用户本人**,或用户显式授权的 IM 用户。不存在"不可信操作者"
3. **用户愿意让 agent 完全代理自己的系统权限**——这是选 Bonsai 的理由

因此 Bonsai **不做**以下设计(都会减弱能力、加摩擦):

- code_run 沙箱(你的机器你做主)
- 浏览器域名限制(无限制访问是核心卖点)
- Prompt injection 防御(会误伤合法用例)
- 破坏性操作默认拦截(默认不加摩擦,`[paranoid]` 段可开)
- 多用户 memory 隔离(单用户工具)

Bonsai **仍然做**(保护用户、不限制能力):

- 密钥不外泄(保护用户资产)
- 外部接口(IM/MCP)的边界认证(防止"别人"用你的 agent)
- 失控保护(保护钱包:max_turns / 重复调用检测 / 预算硬上限)
- 数据可恢复(保护 memory 和文件内容)

详见 [docs/SAFETY.md](docs/SAFETY.md)。**不要按"企业 agent"的安全清单来 review Bonsai**——那不是这个项目的目标。

## 适合谁

**Bonsai 的靶心只有一个**：重度单用户、长周期使用、对成本敏感、希望 agent 随时间"长成自己的样子"——它装在你自己的机器上，只认你一个人，你的 token、你的数据、你说了算。
