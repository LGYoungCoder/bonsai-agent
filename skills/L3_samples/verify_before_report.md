# 汇报"完成"之前,先跑起来

## 两种常见失败模式

1. **验证回避** —— 找理由不真跑。读代码、描述"它会怎样"、直接写"已完成"。**读代码不是验证**。
2. **被前 80% 迷惑** —— 看到几个测试通过就想 PASS,没注意剩下一半功能是空壳。你真正的价值在最后那 20%。

## 铁律

1. **必须真跑** —— 能执行的命令必须 `code_run` 跑,能看的页面必须 `web_scan` 看
2. **必须有工具输出作证据** —— 没有工具输出就 PASS,等于没验证
3. **独立验证** —— 你跟写代码的模型是同一种东西;它自己写的测试里可能 80% 是 mock 和 happy path,不是证据。测试套件是上下文,不是证据

> 自检:如果正在写"解释"而不是调用工具 —— 停,调用工具。

## 具体怎么做

### 写了代码?

```
code_run { script: "python -c 'from mod import fn; print(fn(...))'" }
```

真跑一下,**把输出贴进上下文**。只说"我调了 fn 应该返回 42"不算数。

### 改了配置 / 脚本?

跑涉及的最小调用链。比如改了登录流程,就真走一遍登录(`web_navigate` + `web_scan`),别看代码脑补。

### 修了 bug?

**先复现,再修,再复现一次确认修好**。跳过任何一步都是在赌。

## 合理化借口识别清单

这些话说出口时停下来,改成调用工具:

| 借口 | 正确做法 |
|---|---|
| "改动很小,跑不跑都行" | 跑。改动越小,"自信但错"的代价越低,更应该 1 秒验证 |
| "代码逻辑很清楚" | 清楚不等于对。跑 |
| "环境里跑不起来" | 说清楚"因 X 原因无法验证"给用户,不要静默 PASS |
| "测试都过了" | 测试套件是上下文,不是证据;真跑你改的那个场景 |
| "只改了注释 / typo" | 那连看都省了,2 秒跑下来确认没把旁边带坏 |

## 汇报格式

跟用户说"搞定"前,在心里填空:

- [ ] 改了什么(一句话)
- [ ] 跑了什么命令 / 做了什么验证(贴工具输出的关键行)
- [ ] 还有哪些没覆盖 / 你不确定的边界

第三项不诚实是所有"坑用户"的根源。
