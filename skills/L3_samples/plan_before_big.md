# 动手之前先分步骤

## 触发(必须规划)

满足**任意一条**就先写计划再动手:

- 3 步以上、每步之间有依赖
- 动 3 个以上文件
- 有条件分支(A 通了就做 X,不通做 Y)
- 能并行的任务(并行跑能省时间)
- 用户问"能不能帮我做 XX" 其中 XX 范围模糊

## 跳过(不用规划)

- 单一修改(改 1 个字段、加 1 个打印、问单一事实)
- 用户已经给出明确步骤,你只是执行

## 怎么规划

### 1. 先探,再写

**禁止**在没探清楚现状前就写计划。

探的是三件事:

1. **环境现状** —— 有什么已经在了、什么缺、什么跟假设不一样
2. **可用工具 / SOP** —— `skill_lookup` 看 L1 里有没有现成 SOP 帮上忙
3. **关键不确定点** —— 哪些地方你是在猜(标出来,别藏着)

探的产出是一段 5-10 行的摘要,写进 `update_working_memory`。

### 2. 计划模板

```
目标:一句话

前置:
- [ ] 依赖 A 已装
- [ ] 文件 X 的当前状态是 Y

步骤:
1. 做 P1 → 期望产出 Q1 → 验证 V1
2. 做 P2(依赖 Q1)→ ...
3. 可并行:P3a, P3b

可并行节点(省时间):
- P3a 和 P3b 之间没依赖

不确定 / 风险:
- R1: 如果 A 挂了怎么办 → fallback 是 B
```

### 3. 执行时回看计划

- 每完成一步,心里打勾,**不偏离**。要变更先停下来问"为什么改计划"
- 发现计划错了(常见:探得不够)→ 回到第 1 步重探,**不要**硬推错误计划到底
- 完成一个并行块 → 汇合检查,别让并行的分支静默走歪

## 不要做

- ❌ 脑子里想了个大概就开干 —— 走到一半发现依赖错,回头成本高
- ❌ 写了计划就不回头看 —— 计划是锚,不是圣旨
- ❌ 把规划和探索混在一起 —— 先探,再规,别一边想一边探一边改方案
- ❌ 计划过细(10 步以上,每步 1 行)—— 执行时没用,写也费时间

## 规模参考

| 任务规模 | 规划深度 |
|---|---|
| 1-2 步 | 不写 |
| 3-5 步 | 心里过一遍就开干 |
| 6-10 步 / 跨文件 | 写进 `update_working_memory`(5-10 行) |
| > 10 步 / 跨模块 | 提议开 plan_XXX/ 目录存方案,或者先 `ask_user` 确认范围 |
